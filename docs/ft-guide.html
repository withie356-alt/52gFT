가위바위보 · 자원 · 생일 순 등 팀에서 편한 방식으로 정해주세요. 1일차용입니다!
1 과제 Grouping
문제 리스트 A3 포스트잇

퍼실리테이션

  • 어떤 문제가 많이 나왔는지 파악하는 단계
  • 팀 관심 주제를 구조적으로 살펴보는 단계
  • 포스트잇을 테이블에 펼치고 비슷한 주제끼리 그룹핑
목표: 총 4개 그룹 · 그룹당 1~2개(최대 3개)

예상 질문

그룹은 몇 개 만들어야 하나요?
비슷한 것끼리 묶되, 총 4개 그룹이 적당합니다. 한 그룹에 1~2개(최대 3개).
포스트잇이 두 그룹 사이에 있어도 되나요?
두 그룹에 해당된다고 느끼면 가운데 두고, 팀과 논의해 판단하도록 도와주세요.
사전학습에서 작성한 걸 다 가져와야 하나요?
네, 1인당 1개이므로 모두 펼쳐주세요. 안 가져온 팀원은 현장에서 보충 가능.
2 과제 1차 선정 (멀티보팅)
10분 스티커

퍼실리테이션

  • 1인당 스티커 2장으로 투표 (기준 활용해 붙이기)
  • 동점 시 "어떤 과제가 우리 팀에 더 중요한가요?" 질문으로 합의 유도

예상 질문

스티커 2개를 하나의 포스트잇에 붙여도 되나요?
1장의 포스트잇에는 1개의 스티커만. 스티커 갯수로 몇 명이 동의했는지 파악하기 위함입니다.
동점이 나왔을 때는?
동점끼리 재투표, 혹은 논의로 결정. "혹시 이 아이디어를 꼭 하면 좋겠다 생각하신 분부터 의견 들어볼까요?" 후 재투표도 방법.
3 과제 오너 인터뷰
포스트잇 네임펜

퍼실리테이션

  • 목표: 과제 오너의 실제 경험에서 숨은 페인포인트 발견

예상 질문

인터뷰는 누가 하나요?
과제 오너가 인터뷰이, 나머지가 인터뷰어로 질문. 오너 부재 시 경험을 더 잘 이야기할 수 있는 팀원을 내부 선정.
시간이 부족하면?
대형 워크숍이라 시간 준수가 핵심. 10분 남음 → 5분 → 3분 전 "마지막 질문해주세요" 순서로 안내.

참고사항

  • 확보할 것 — 페르소나 · 행동 단계 · Painpoint + 감정 · 이해관계자
    · 행동 단계 = "이 일을 어떤 순서로 하세요?"로 물어보고 오너의 답을 정리 (→ 다음 스텝 여정지도 4단계 재료). 예: 피드백 수집 → 분류 → 팀 공유 → 반영
  • 좋은 질문 — 스토리와 근본 원인을 끌어내기
    · "최근 ○○ 했을 때 처음부터 끝까지 얘기해주세요" + "왜요?" 3~5회
  • 피할 질문 — 답 유도 · 솔루션 묻기 · 추상적 질문
    · "불편하셨죠?" / "어떤 기능 있으면?" / "평소에 어떠세요?"
4 페르소나 프로필 정의
페르소나 A4

퍼실리테이션

  • 목표: 팀의 공통 고객 인물을 구체화 · Align
  • 페르소나는 가상 인물이지만 인터뷰 데이터 기반이어야 함
  • 여러 페르소나가 나오면 하나를 선택하도록 가이드

예상 질문

완전히 가상으로 만들어도 되나요?
가상이지만 인터뷰에서 나온 실제 데이터에 기반해야 합니다. 상상만으론 이후 아이디에이션 판단이 어려워요.
팀에 페르소나가 여러 개라면?
가급적 1명으로 정의. 이 단계에선 1개, 추후 이해관계자별 POV를 작성할 예정.

참고사항

  • 구성 요소 — 이름·직무 / 업무 맥락 / 목표 / Painpoint / 대표 말
  • 예시 — 김민정, 32세, 스타트업 PM · 고객 피드백이 Intercom·슬랙·노션에 흩어져 우선순위 판단이 느림 · "비슷한 피드백을 세 군데서 받는데 한 사람인지 여러 명인지 모르겠어요."
  • POV 미리보기 — Point of View. 페르소나 관점에서 문제를 한 문장으로. 형식: [페르소나]는 [니즈]가 필요하다, 왜냐하면 [인사이트]이기 때문. 본격 작성은 STEP 8.
5 여정지도 (4단계) 정의
이젤패드 노이란트 펜

퍼실리테이션

  • 목표: 페르소나 Needs/Painpoint를 구조적으로 발산하도록 돕기

예상 질문

4단계는 어떻게 나누나요?
페르소나가 과제를 충족시키기 위해 거치는 단계. 예: 발견 > 제안확인 > 팀논의 > 활용
여정단계를 구체적으로 어떻게?
과제 유형에 따라 자유롭게 업무주제/구매/서비스 이용 등 리얼한 단계로. 추상적 단어보다 페르소나의 실제 행동과 가까운 단어로.

참고사항 · 주제별 예시

  • 현장 공정 관리 — 공정 계획 → 자재 수급 → 시공·감리 → 준공 검수
  • 편의점 발주·재고 관리 — 재고 확인 → 발주 → 입고·진열 → 판매 분석
  • 주유소 고객 서비스 — 차량 진입 → 주유 → 결제 → 부가 서비스
  • 호텔 고객 체크인 — 예약 확인 → 체크인 → 객실 안내 → 체크아웃
  • 발전소 운영 — 연료 수급 → 발전 → 송·배전 → 실적 관리
6 여정지도 내 Painpoint 발산
단계별 5개 · 총 20개

퍼실리테이션

  • 목표: 인터뷰 데이터 기반 페르소나 단계별 Needs/Painpoint 발산
  • 모두가 5개 채울 수 있도록 확인과 질문

예상 질문

20개 다 채워야 하나요?
단계별 5개씩 총 20개 채우도록 독려. 시간 부족 시 각자 한 단계를 맡아 5개 작성 후 같이 리뷰도 OK.
나와 중복되는 내용은?
중복돼도 괜찮습니다. 비슷하지만 다르게 느껴지는 건 구분할 수 있도록 의견을 짧게 주면 좋습니다.

참고사항 · FT 조력 포인트

  • Pain 찾을 포인트 — 시간 지연 구간 · 대기·병목 · 같은 작업 반복 · 수기 입력 · 업무 중단 · 컨텍스트 스위칭 · 실수·누락 · 재작업 · 승인·응답 대기 · 부서 간 핸드오프 마찰 · 도구 전환·기능 한계 · 감정(짜증·답답·막막·포기·불안)
  • Need 찾을 포인트 — 자동 처리·알림 · 한 곳에서 통합 보기 · 실시간 파악·감지 · 단계 축소·입력 최소화 · 진행 상황 가시화·근거 추적
  • 발산 유지 — 감정 단어 권장 / 평가·해결책 들어가면 "지금은 발산 시간"으로 차단
7 핵심 Painpoint 선정
스티커 의사결정매트릭스 A4

퍼실리테이션

  • 목표: 4가지 기준으로 핵심 Painpoint 1개 선정
  • 멀티보팅으로 4개 수렴 → 의사결정매트릭스로 선정
  • 동점 시 "어느 Painpoint가 페르소나에게 해결되어야 할 문제인가요?"
  • 1개(개수)에 집중하기보다 문제를 정확히 정의하는 게 중요
  • 너무 지엽적이면 묶어서 해결할 만한 크기로 가이드

예상 질문

선정 문제가 "우리만의 노력"으로 해결되기 어렵다면? (예: 과도한 예산, 법규 개정)
2일 안에 해결책 찾기 어렵다면 다른 문제 제안도 방법. 짧은 시간에 결과물을 만들 수 있는 문제가 좋아요.

참고사항 · 선정된 Pain의 위치

  • 이 단계 선정 Pain은 전체의 출발점 — "어떤 분의 어떤 문제를 풀었다"의 바로 그 "문제"가 됨
  • 편의점 점주: 재고 수기 확인에 매일 20분, 마감 시간에 판단 실수
  • 호텔 프런트: 예약·고객 정보가 여러 시스템에 나뉘어 체크인 응대가 지연됨
  • 주유소 직원: 단골 차량 이력이 파악 안 돼 맞춤 서비스·업셀링 기회 놓침
  • 건설 현장 PM: 자재 입고·공정 변경이 팀·협력사에 실시간 전달 안 돼 대기 발생
  • 발전소 운영자: 설비 이상 징후가 정기 점검 후에야 드러나 대응 지연
8 POV를 활용한 문제정의
POV 툴킷

퍼실리테이션

  • 목표: POV 프레임워크로 명확하게 문제 정의

참고사항 · Pain → POV 변환 예시

  • 공식 — [페르소나]는 [니즈]가 필요하다. 왜냐하면 [인사이트]이기 때문이다.
  • 편의점 점주 — "점주는 재고를 한눈에 알 방법이 필요하다. 왜냐하면 마감에 쫓겨 판단 실수가 반복되기 때문이다."
  • 호텔 프런트 — "프런트는 예약·고객 정보를 한 화면에서 볼 방법이 필요하다. 왜냐하면 정보가 여러 시스템에 나뉘어 체크인이 지연되기 때문이다."
  • 주유소 직원 — "직원은 단골 차량 이력을 즉시 확인할 방법이 필요하다. 왜냐하면 이력을 모르면 맞춤 서비스와 업셀링 기회를 놓치기 때문이다."
  • 건설 현장 PM — "PM은 자재·공정 변경을 팀·협력사에 실시간 공유할 방법이 필요하다. 왜냐하면 변경 전달 지연이 전체 공정 대기로 이어지기 때문이다."
  • 발전소 운영자 — "운영자는 설비 이상 징후를 조기에 감지할 방법이 필요하다. 왜냐하면 정기 점검 때에야 드러나 대응이 늦어지기 때문이다."
  • 팁 — 페르소나는 구체적으로 · 니즈는 솔루션 ✗ 목적 ✓ · 인사이트는 "왜 중요한가"
9 이해관계자별 POV 작성
POV 툴킷

퍼실리테이션

  • 목표: 다양한 이해관계자 관점에서 POV 작성 → 최적 문제정의 선택

예상 질문

모든 이해관계자를 작성해야 하나요?
최대 3명 정도가 적당. 개수가 많으면 아이디에이션 연결 판단이 어려워요.

참고사항 · 이해관계자별 POV 예시

  • 예 1 · 편의점 재고 확인 Pain
    · 점주: "점주는 재고를 한눈에 알 방법이 필요하다. 마감에 쫓겨 판단 실수가 반복되기 때문."
    · 본사 MD: "MD는 점포 재고·판매를 실시간 파악할 방법이 필요하다. 점주 감에 의존한 발주가 전체 최적화에 한계가 있기 때문."
    · 방문 고객: "고객은 원하는 상품이 품절되지 않을 환경이 필요하다. 품절 반복이 브랜드 충성도를 떨어뜨리기 때문."
  • 예 2 · 발전소 설비 이상 Pain
    · 운영자: "운영자는 이상 징후를 조기 감지할 방법이 필요하다. 정기 점검 이전엔 드러나지 않기 때문."
    · 본사 엔지니어링팀: "팀은 여러 발전소 상태를 통합 모니터링할 방법이 필요하다. 개별 보고에 의존하면 전사 리스크 판단이 어렵기 때문."
    · 전력 공급 책임자: "책임자는 예측 가능한 안정 가동 데이터가 필요하다. 예기치 않은 설비 이상이 공급 계약 신뢰성을 흔들기 때문."
  • 다른 케이스 이해관계자 예
    · 호텔 프런트 체크인 지연 Pain — 프런트 직원 / 투숙 고객 / 호텔 매니저
    · 주유소 고객 이력 미파악 Pain — 주유소 직원 / 단골 고객 / 주유소 점장
    · 건설 공정 변경 전달 지연 Pain — 현장 PM / 협력사 / 본사 공사관리팀
  • 왜 여러 관점? — 같은 Pain도 관점에 따라 니즈·인사이트가 달라짐. 3명 비교 후 가장 본질적인 관점을 선택하면 해결 범위·파급력이 넓어짐
10 POV별 Accept 여부 판단

퍼실리테이션

  • 목표: 작성된 이해관계자별 POV 중 중요하면 수용
  • 이후 아이디에이션 과정에서 관점으로 활용 예정

예상 질문

Accept/Reject 기준?
① 단일 관점으로 해결해도 괜찮은지? 이해관계자 관점을 고려해야 더 좋은 솔루션이 나오는지?
② 복수 관점이 모두 해결돼야 진짜 문제 해결에 가까운지?
③ 우리 팀 과제와 관련 있는지?
모두 Yes → Accept, NO가 많다면 Reject 고려.

참고사항 · Accept/Reject 예시 (편의점 재고 Pain)

  • 점주 POV — Accept (페르소나 본인, 아이데이션 중심축)
  • 본사 MD POV — Accept — 점주와 다른 층위에서 독립 가치, 데이터 기반 발주 최적화 솔루션으로 연결 가능
  • 방문 고객 POV — Reject — 해결 방향이 점주 POV와 겹치고(품절 최소화=결국 재고 관리), 2일 내 다루기엔 범위가 너무 넓음
  • 팁 — 보통 페르소나 POV + 이해관계자 1~2개 정도 Accept가 아이데이션 생산적
11 POV 체크리스트 (검증)

퍼실리테이션

  • 목표: 선정한 POV의 완성도를 체크리스트로 검증

예상 질문

체크리스트 항목이 많아서 어려워요.
핵심은 "POV가 우리가 해결하려는 Painpoint를 명확히 표현하는가?". 이 목적만 달성되면 다른 체크리스트는 가볍게 살펴봐도 좋아요.

참고사항 · POV 체크리스트 7문항

  • 1. 페르소나가 구체적인가? — 일반론 ✗ (예: "40대 팀장, 신규 PJT 리더" ✓)
  • 2. 니즈가 목적인가? — 솔루션·기능 ✗ ("앱이 필요" ✗ / "빠르게 판단할 방법" ✓)
  • 3. 인사이트가 "왜 중요한가"인가? — 단순 증상 나열 ✗ · 근본 이유·영향 ✓
  • 4. 인터뷰 데이터로 뒷받침되나? — 상상만으로 ✗
  • 5. Painpoint를 명확히 표현하나? — 핵심 기준 (Q&A 참조)
  • 6. 2일 내 다룰 크기인가? — 너무 넓지도 좁지도 않게
  • 7. HMW가 자연스럽게 나오나? — 다음 단계 연결성 (예: "어떻게 하면 빠르게 판단할 수 있을까?")

1 [아이데이션 1단계] HMW 만들기
인원수만큼

퍼실리테이션

  • 목표: 문제정의(POV)를 아이데이션 질문(HMW)으로 전환
  • HMW 공식: "어떻게 하면 ~할 수 있을까?"

예상 질문

HMW를 몇 개 써야 하나요?
4인팀은 4개, 5인팀은 5개로 합의. 브레인라이팅 때 한 장당 하나의 HMW에서 돌아가며 아이디어를 내야 하므로 인원수대로 필요합니다.

참고사항 · POV → HMW 변환 예시

  • 편의점 점주 — "어떻게 하면 점주가 재고를 한눈에 알 수 있을까?"
  • 호텔 프런트 — "어떻게 하면 프런트가 예약·고객 정보를 한 화면에서 볼 수 있을까?"
  • 주유소 직원 — "어떻게 하면 직원이 단골 차량 이력을 즉시 확인할 수 있을까?"
  • 건설 현장 PM — "어떻게 하면 PM이 자재·공정 변경을 팀·협력사에 실시간 공유할 수 있을까?"
  • 발전소 운영자 — "어떻게 하면 운영자가 설비 이상을 조기에 감지할 수 있을까?"
  • 크기 팁 — 너무 넓으면 발산 흐릿 · 너무 좁으면(버튼 색 수준) 아이디어 금방 고갈
2 [아이데이션 2단계] 브레인라이팅
브레인라이팅 A3

퍼실리테이션

  • 목표: 최대한 많은 아이디어를 발산
  • 한 사람당 1개의 HMW에 3개 이상 아이디어 제시
⚠️ 평가 · 조언 · 구체적 대안 금지. 발산에만 집중!

예상 질문

평가하거나 조언하거나 구체적 대안을 대화하고 있다면?
지금 단계는 양보단 질입니다. 중단 요청.
예: "지금은 발산 시간이니, 좀 더 아이디어를 발산해볼까요?"

참고사항 · FT 조력 포인트

  • 발산 자극 각도 — 기존 도구 응용 · 타 업계 사례 차용 · 서로 다른 아이디어 결합 · 역발상(반대로 하면?) · 사람·장소·시간 바꾸기 · 기술(AI·센서·앱) 얹기
  • FT 개입 멘트 — 평가·조언 들어가면 "지금은 발산 시간, 대안 토론은 잠시 후에" · 침묵 길어지면 "이거 살짝 비틀면 뭐가 될까요?" · 비슷한 것만 나오면 "정반대로 하면?"
  • 양 목표 — HMW 한 장당 12~15개는 나와야 다음 투표가 의미 있음. 막히면 라운드 로테이션으로 재시작
3 브레인라이팅 중 아이디어 투표
스티커

퍼실리테이션

  • 목표: 아이디어 수렴 (HMW당 1개 아이디어 선정)

예상 질문

몇 개 투표해야 하나요?
한 장당 12~15개 아이디어. 한 장당 4개씩 멀티보팅. (Main FT 가이드에 따라 달라질 수 있음)

참고사항

  • 투표 기준 예 — 페르소나 임팩트(Pain 해결도) · 참신함 · 2일 내 실행 가능성
  • 동점 중재 질문 — "페르소나 입장에서 어느 게 더 매력적인가요?" · "쇼케이스 때 어떤 게 더 장면이 그려지나요?"
4 [아이데이션 3단계] SCAMPER
선정된 아이디어

퍼실리테이션

  • 목표: SCAMPER로 선정 아이디어에 새로운 관점 더하기

예상 질문

SCAMPER 7가지 다 해야 하나요?
아니에요. 1~2가지 중 "지금 말한 아이디어에서 출발해 떠오르는 것"만 선택해 집중하면 됩니다.

참고사항 · SCAMPER 7가지

  • S · Substitute (대체) — 다른 걸로 바꾸면? (예: 수기 → 센서)
  • C · Combine (결합) — 다른 것과 합치면? (예: POS + 재고 알림)
  • A · Adapt (응용) — 다른 분야 사례 차용 (예: 배달앱 실시간 추적을 공정에)
  • M · Modify (변경) — 크기·형태·속도 바꾸면? (예: 하루 보고 → 실시간)
  • P · Put to other use (전용) — 다른 용도로? (예: 고객 QR을 점주 재고에)
  • E · Eliminate (제거) — 없애면? (예: 수기 장부 완전 폐지)
  • R · Reverse (역발상) — 반대로 하면? (예: 점주가 아닌 본사가 먼저 알림)
  • 활용 팁 — 전부 하지 말고 2~3개만 골라 아이디어 자극용으로
5 아이디어 스크리닝 보드
이젤패드

퍼실리테이션

  • 목표: HMW 선정 아이디어 + SCAMPER 아이디어 중 1~3개 수렴

예상 질문

1개로 수렴 vs 2~3개 수렴 차이?
2~3개라면 "어떻게 하나로 연결할 수 있나?" 생각해 하나로 연결하거나, 1개의 큰 아이디어에 세부 3가지 기능으로 도출 가능. 오늘 프로토타입 시간 활용해 결과물 낼 수 있도록 우선순위/좁히기를 제안해요.

참고사항

  • 수렴 기준 — 페르소나 Pain 해결도 · 2일 내 프로토타입 가능성 · 쇼케이스 장면이 그려지는가
  • 1개 수렴 — 하나 깊게 파기 (프로토타입 완성도 ↑)
  • 2~3개 수렴 — 큰 아이디어 1개 + 세부 기능 2~3개로 재구성 추천
6 아이디어 컨셉보드
컨셉보드 A3

퍼실리테이션

  • 목표: 수렴 아이디어를 컨셉보드에 정리 → 생각 합의(정리)
  • 1~3개로 수렴된 아이디어를 3개의 세부 아이디어로 나눠 정리

예상 질문

6컷을 어떻게 작성하고 무슨 내용을 써야 하나요?
컨셉보드 아이디어를 실제 사용자가 어떤 흐름으로 이용하게 되는지 구성하는 것이 중요.
그림을 꼭 그려야 하나요?
완성도 높은 그림보다 사용자가 어떻게 활용하는지 Flow 연결이 중요. 글로 표현하다 어려운 부분만 그림으로 보완해도 OK.

참고사항 · 6컷 구성

  • 플로우 흐름 예 — ① 언제·어디서(상황) → ② 페르소나 행동 → ③ 시스템 반응 → ④ 문제 해결 순간 → ⑤ 결과·감정 → ⑥ 확장·다음 행동
  • 예: 편의점 점주 — 마감 시간 도착 → 대시보드 확인 → 부족 상품 자동 표시 → 원탭 발주 → 재고 확인 완료 → 다음 날 원활 운영
  • 핵심 — 그림 완성도 ✗ / 사용 맥락 연결 ✓ (쇼케이스 때 1분 안에 설명 가능해야)

6컷 스토리보드

퍼실리테이션

  • 목표: 아이디어를 구체적으로 정의해 오후 세션(프로토타입)으로 연결
  • 아이디어 활용 플로우를 구현하며 세부 기능 도출

예상 질문

어느 정도로 구체적으로 적어야 하나요?
세부적으로 어떤 기능이 있어야 잘 작동하는지를 중심으로. 버튼 모양/색깔 정의보단 버튼이 필요한 이유가 명확해야 추후 바이브 코딩 때 의도한 바를 구현할 수 있어요.

참고사항

  • 시나리오 구조 — 언제·어디서(상황) · 페르소나 행동 · 시스템 반응 · 결과·감정
  • 기능 정의 팁 — 각 기능마다 "이 기능이 왜 필요한지" 한 줄로 설명 가능해야 프로토타입 단계에서 흔들리지 않음
  • 예: 재고 알림 기능 — "마감 1시간 전 자동으로 부족 상품 리스트를 띄워 점주가 놓치지 않게" (버튼 색·위치는 프로토타입이 정함)
  • 프로토타입 연결 — 바이브 코딩/MISO 코딩에이전트가 이해할 수 있도록 동사 중심으로 (감지한다 · 알린다 · 묶는다 · 보여준다)
